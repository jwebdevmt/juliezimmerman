Migrating Real Estate Data Pipelines from RETS to RESO Web API

The real estate industry spent years running on RETS — the Real Estate Transaction Standard — a protocol that did its job but was never designed for the web-first world that eventually arrived. RESO Web API is its replacement: RESTful, JSON-based, and built around modern HTTP conventions. The migration from one to the other sounds straightforward. In practice, it requires rethinking how your data pipeline works at every layer.

I've managed this migration for multiple real estate clients, each with different MLS relationships, different data volumes, and different display requirements. Here's what I've learned about where the real complexity lives.

The first thing to understand is that RETS and RESO Web API don't just differ in protocol — they differ in data model. RETS returns data in a proprietary format with field names that vary by MLS. RESO standardizes field names through the Data Dictionary, but the actual coverage of that standard varies significantly by MLS. Some have adopted it fully. Others have partial implementations. You need to map each client's actual field output to what your system expects, and that map is rarely clean.

Photo ingestion is where most migrations get complicated. RETS delivered photos through a separate media endpoint with its own quirks. RESO Web API standardizes this through Media resources, but the practical behavior — rate limits, batch sizes, retry behavior — varies enough by MLS that you can't assume a single approach works everywhere. I've built separate ingestion logic for clients on different MLSs even when they're both theoretically on RESO Web API.

Cron architecture matters more than most people expect. High-volume listing pipelines need to respect API throttle limits while still keeping data current. The naive approach — poll frequently and back off when throttled — produces inconsistent data freshness and wastes request budget. A better approach is to design the sync around delta detection: only pull records that have actually changed since the last successful sync, and build the local data model to track enough state to know what's current without asking the API.

The migration itself is best handled as a parallel run. Keep the RETS pipeline running while you build and validate the RESO Web API pipeline against the same data. When the outputs match consistently, you cut over. This costs more time upfront but eliminates the risk of discovering a mapping gap after you've already turned off the old system.

What looks like a protocol upgrade is really a data architecture project. The clients who handle it smoothly are the ones who understand that before the first line of code changes.
